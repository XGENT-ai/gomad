## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Repo → CI | `src/domain-kb/**` author-committed content flows through `npm run quality` in CI; un-attributed content must be blocked before merge. |
| Validator → parsed YAML | `yaml.parse` on untrusted frontmatter (author-controlled) — low-trust but not adversarial (author is the repo maintainer; KB content is not end-user input). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01-01 | Tampering (I) | tools/validate-kb-licenses.js frontmatter parser | mitigate | Use `yaml.parse` (battle-tested library) rather than hand-rolled regex; catch parse exceptions and report as CRITICAL finding (see `extractFrontmatter` in action). LOW severity: KB frontmatter is author-committed, not user input. |
| T-10-01-02 | Information Disclosure (I) | KB content without attribution | mitigate | KB-02/KB-03 rules block merge if `source:` / `license:` missing. CRITICAL severity per PITFALLS.md §4-J — legal risk if violated. |
| T-10-01-03 | Elevation of Privilege (E) | Release gate bypass | mitigate | `validate:kb-licenses` wired into `npm run quality` chain via Task 2; no separate script that could be skipped. Acceptance criteria include position verification to prevent accidental re-ordering that moves it out of the chain. CRITICAL severity: bypassing gate means un-attributed content ships. |
| T-10-01-04 | Denial of Service (D) | Validator performance | accept | KB tree is small (~20 files max v1.3 per D-12). `globSync` + per-file parse is O(N); no DoS surface for CI. |
| T-10-01-05 | Repudiation (R) | Missing `last_reviewed` date | mitigate | KB-04 rule requires `YYYY-MM-DD` date; enables audit of content freshness. MEDIUM severity: staleness is a review, not a ship-blocker. |
