       - You're fixing a bug in the installer, in a tool under `tools/`, or in an existing skill.
       - You're improving documentation or translations.

    3. `## Prerequisites` — admonition:
       ```markdown
       :::note[Prerequisites]
       Node.js 20+, npm 10+, a GitHub account, and read access to the existing `CONTRIBUTING.md` at the repo root for the canonical PR template.
       :::
       ```

    4. `## Step 1: Fork and clone` — 2 short paragraphs + code block:
       - Direct contributors to fork `xgent-ai/gomad` on GitHub then clone the fork:
         ```bash
         git clone git@github.com:<your-username>/gomad.git
         cd gomad
         npm install
         ```
       - Note: GoMad is a hard fork of BMAD Method (not continuously merged downstream).

    5. `## Step 2: Branch from main` — 1 paragraph + code:
       ```bash
       git switch -c feat/your-feature-name
       ```
       - Naming convention: `feat/`, `fix/`, `docs/`, `chore/` prefixes.

    6. `## Step 3: Make your change` — 2-3 paragraphs:
       - Source layout overview: source-of-truth content lives under `src/gomad-skills/`, `src/core-skills/`, and `src/domain-kb/`. Installer copies these into the user's `<installRoot>/_config/` at install time (the generate-don't-move pattern).
       - For a new persona: add a `src/gomad-skills/<phase>/gm-agent-<short>/` directory with `SKILL.md` (frontmatter `name:` + `description:`) and `skill-manifest.yaml` (`displayName`, `title`, `icon`).
       - For a new task skill: same dir layout under the appropriate phase, with `SKILL.md` only (no skill-manifest.yaml needed for task skills).
       - For docs changes: edit files under `docs/`. The Starlight site picks up changes automatically.
       - Cross-link in prose: "See the [architecture explainer](../explanation/architecture.md) for how installer + workflow + launcher fit together."

    7. `## Step 4: Run the quality gate` — 2 paragraphs + code:
       ```bash
       npm run quality
       ```
       - Explain what this runs (per package.json line 56): format check, ESLint, markdownlint, docs build (link-check + Astro build), install/integration tests, file-reference validation, skills validation, inject-reference-tables test, KB license validation, orphan-reference test.
       - The quality gate must exit 0 before opening a PR.
       - Admonition for when something fails:
         ```markdown
         :::caution[If a check fails]
         Run the failing script in isolation (e.g., `npm run lint` or `npm run docs:build`) for focused output. Don't bypass the gate — the PR template asks you to confirm `npm run quality` passes locally.
         :::
         ```

    8. `## Step 5: Open the PR` — 2 paragraphs:
       - Push your branch and open a PR against `main` of `xgent-ai/gomad`.
       - PR title convention: `<type>(<scope>): <summary>` (e.g., `feat(skills): add gm-domain-skill`).
       - PR body should reference any related issue or roadmap entry.

    9. `## What you get` — bullet list:
       - PR review feedback (typically within a few days).
       - Once merged, your change ships in the next `@xgent-ai/gomad` npm release per the milestone schedule.

    10. `## Where to learn more` — single paragraph (prose, NOT a "Next:" header):
        - Read the [architecture explainer](../explanation/architecture.md) for the runtime model.
        - Browse [agents reference](../reference/agents.md) and [skills reference](../reference/skills.md) for catalog examples.
