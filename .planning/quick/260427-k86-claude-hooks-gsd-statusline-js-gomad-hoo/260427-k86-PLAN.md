# Quick Task 260427-k86 — gomad Claude Code statusline + installer wiring

## Goal

Ship a gomad-aware Claude Code statusline:

```
<model> │ 👤 Amelia (gm-agent-dev) · ▶1 ✓3 ⏳5  │ <dirname> ████░░░░░░ 38%
```

…and make `gomad install` (claude-code platform) deploy it automatically: copy the JS into `.claude/hooks/`, register it under `statusLine` in `.claude/settings.json`, and unwind both on `gomad uninstall`.

Out of scope: deploying for IDEs other than claude-code (only claude-code has a native statusline contract); a real-time agent-detection daemon (statusline runs once per Claude Code refresh, no daemon needed).

## Tasks

### Task 1 — Author `tools/installer/assets/hooks/gomad-statusline.js`

**Files:** `tools/installer/assets/hooks/gomad-statusline.js` (new)

**Action:**

Create the new statusline. Behavior, patterned on `.claude/hooks/gsd-statusline.js`:

- Shebang `#!/usr/bin/env node`. Pure Node, no third-party deps.
- Read JSON from stdin with the same 3 s timeout guard as the reference. Silent-fail on parse errors.
- Extract: `model.display_name`, `workspace.current_dir`, `session_id`, `context_window.remaining_percentage`, `context_window.total_tokens`.
- **Context bar:** reuse the GSD reference's progress-bar logic verbatim (10-segment █/░ bar, AUTO_COMPACT_BUFFER_PCT, color tiers green<50, yellow<65, orange<80, blink-red≥80, env override `CLAUDE_CODE_AUTO_COMPACT_WINDOW`). Skip the GSD bridge-file write — that's GSD-internal.
- **Sprint snapshot resolver** (`readGomadSprint(dir)`):
  1. Walk up from `dir` (max 10 levels, stop at `$HOME` or filesystem root) looking for `_gomad/agile/config.yaml`.
  2. If found, parse the YAML to read `implementation_artifacts` (resolve `{project-root}` placeholder against the dir holding `_gomad/`).
  3. Read `<implementation_artifacts>/sprint-status.yaml`. Parse top-level `project`, then walk `development_status` map and tally story statuses (skip keys starting with `epic-`, skip keys ending with `-retrospective`). Map legacy `drafted` → `ready-for-dev`.
  4. Return `{ project, counts: { backlog, ready_for_dev, in_progress, review, done } }` or `null` if any step fails.
  - **YAML parsing:** keep it dependency-free. Use a tiny line-by-line parser scoped to what we actually need (top-level `key: value` for the metadata block, then a flat `key: value` walk under `development_status:` 2-space indented children). Reject unparseable structure silently and return `null`.
- **Agent persona detector** (`detectGomadAgent(taskText)`):
  - Hard-coded map: `{ Mary: 'gm-agent-analyst', John: 'gm-agent-pm', Winston: 'gm-agent-architect', Sally: 'gm-agent-ux-designer', Amelia: 'gm-agent-dev', Bob: 'gm-agent-sm', Barry: 'gm-agent-solo-dev', Paige: 'gm-agent-tech-writer' }`.
  - First match the skill id (`gm-agent-*`) — anchored, so `gm-agent-dev` doesn't match `gm-agent-developer`. Fall back to whole-word persona-name match (case-sensitive on the first letter to dodge "amelia earhart"–style false positives in unrelated todos).
  - Return `{ persona, skill }` or `null`.
- **Todo lookup:** identical to the reference (`CLAUDE_CONFIG_DIR ?? ~/.claude` → `todos/`, find newest `<session>-agent-*.json`, pick first `status === 'in_progress'`'s `activeForm`). Then run the activeForm string through `detectGomadAgent` to label the active persona.
- **Output composition** — three info zones with `│` dividers. If a zone is empty, drop it (and its divider) entirely:
  - `model` (always shown, dimmed)
  - middle: `👤 Amelia (gm-agent-dev)` if persona detected, else the bare todo `activeForm` (bold), else sprint summary `Sprint: <project> · ▶<in_progress> ✓<done> ⏳<backlog+ready_for_dev>` (dim) when sprint state was resolvable, else nothing.
  - `dirname` (dimmed) + `ctx` bar
- Module footer: `module.exports = { readGomadSprint, detectGomadAgent };` plus `if (require.main === module) runStatusline();` so unit tests can require helpers without invoking stdin.
- Keep total file size under ~200 LOC. Prioritize matching the reference's silent-fail discipline — the statusline must NEVER throw or hang.

**Verify:**
- `node tools/installer/assets/hooks/gomad-statusline.js < /dev/null` exits 0 within 4 s.
- `echo '{"model":{"display_name":"Sonnet 4.6"},"workspace":{"current_dir":"'"$PWD"'"},"context_window":{"remaining_percentage":80,"total_tokens":1000000}}' | node tools/installer/assets/hooks/gomad-statusline.js` prints a line containing `Sonnet 4.6` and a `%` token.
- `node -e 'const {detectGomadAgent} = require("./tools/installer/assets/hooks/gomad-statusline.js"); console.log(detectGomadAgent("gm-agent-dev: implement the login form"))'` prints `{ persona: 'Amelia', skill: 'gm-agent-dev' }`.

**Done:** File exists, executable bit set (`chmod +x`), exports the documented helpers, all three verify lines pass.

### Task 2 — Wire claude-code installer to deploy hook + register statusLine

**Files:**
- `tools/installer/ide/platform-codes.yaml` (modified)
- `tools/installer/ide/_config-driven.js` (modified — both `installToTarget` and `cleanup`)
- `package.json` (modified — `files` allowlist)

**Action:**

1. **platform-codes.yaml** — extend the `claude-code` block with two new keys:
   ```yaml
   claude-code:
     ...
     installer:
       ...
       hooks_target_dir: .claude/hooks
       statusline:
         source: tools/installer/assets/hooks/gomad-statusline.js
         dest_name: gomad-statusline.js
         settings_file: .claude/settings.json
   ```
   These keys are claude-code-only; other platforms continue to ignore them.

2. **`_config-driven.js` — `installToTarget`:**
   - After the existing skill + launcher install blocks, add a guarded block: `if (this.installerConfig.statusline) { await this.installStatusline(projectDir, options); }`.
   - New method `installStatusline(projectDir, options)`:
     - Resolve `gomadSrcRoot` via `getProjectRoot()` (already imported indirectly through `path-utils` — add a fresh `require('../project-root')` if needed).
     - Source: `path.join(gomadSrcRoot, statusline.source)`. Skip silently if the source file doesn't exist (graceful for from-tarball edge cases).
     - Dest dir: `path.join(projectDir, hooks_target_dir)` — `fs.ensureDir`.
     - Dest file: `path.join(destDir, statusline.dest_name)` — `fs.copy(src, dest, { overwrite: true })`, then `fs.chmod(dest, 0o755)`.
     - Track via `options.trackInstalledFile?.(destFile)` for manifest correctness.
     - Settings merge: read `<projectDir>/<settings_file>` if it exists, else start with `{}`. Parse JSON; on parse failure, log a warning via `prompts.log.warn` and **abandon the merge** (do not corrupt the file). Set `parsed.statusLine = { type: 'command', command: 'node "$CLAUDE_PROJECT_DIR"/' + hooks_target_dir + '/' + dest_name, padding: 0 }`. Write back with `JSON.stringify(parsed, null, 2) + '\n'`. Track the settings file too.
     - On success print one info line: `Claude Code statusline installed → <hooks_target_dir>/<dest_name>` (respect `options.silent`).
   - Pre-flight an "is this our statusLine?" check before overwriting: if `parsed.statusLine?.command` exists and does NOT contain `gomad-statusline.js`, leave it untouched and warn — do not stomp third-party statuslines.

3. **`_config-driven.js` — `cleanup`:**
   - Add a parallel guarded block at the **start** of cleanup (mirroring the `legacy_targets` and copilot/kilo/rovo blocks): `if (this.installerConfig?.statusline && this.name === 'claude-code') { await this.cleanupStatusline(projectDir, options); }`.
   - New method `cleanupStatusline`:
     - Remove `<projectDir>/<hooks_target_dir>/<dest_name>` if present (`fs.remove`).
     - Read settings file; if `parsed.statusLine?.command?.includes('gomad-statusline.js')`, delete `parsed.statusLine` and rewrite. If `parsed` becomes `{}` and the file was created by us (no other top-level keys before our edit — best-effort: just leave the empty object in place), keep the file — Claude Code tolerates empty objects.
     - If `<hooks_target_dir>` is now empty after the rm, `fs.rmdir` it (best-effort, swallow errors).
   - Important: do NOT touch `statusLine` if it doesn't reference our file — same defensive rule as install.

4. **package.json** — add `"tools/installer/assets/"` to the `files` array so the npm package ships the new asset. Place it lexicographically near the existing `tools/installer/` entry.

**Verify:**
- `node tools/installer/gomad-cli.js install --help` still exits 0 (sanity — no syntax error in the modified modules).
- Hand-run the install path against a temp dir (covered by Task 3's test).

**Done:** All three files modified, `node -c tools/installer/ide/_config-driven.js` parses, `node -e "require('./tools/installer/ide/_config-driven.js')"` loads cleanly.

### Task 3 — Smoke test for the statusline install round-trip

**Files:** `test/test-statusline-install.js` (new), `package.json` (modified — wire into `npm test`)

**Action:**

Style-match the existing `test/test-installation-components.js` (plain Node, no test framework — `assert` from `node:assert/strict`, top-level async IIFE, `process.exit(1)` on failure).

The test should:
1. Create a tmp project dir (`fs.mkdtemp(os.tmpdir() + '/gomad-statusline-')`).
2. Construct a fake `installerConfig` matching the new `claude-code` shape and instantiate `ConfigDrivenIdeSetup` directly.
3. Stub `getProjectRoot()` to point at the gomad repo root (so `tools/installer/assets/hooks/gomad-statusline.js` resolves).
4. **Install case A — no pre-existing settings.json:**
   - Run `setup(tmpDir, ...)`. Assert `<tmp>/.claude/hooks/gomad-statusline.js` exists, mode includes `0o100` (executable). Assert `<tmp>/.claude/settings.json` parses to `{ statusLine: { type: 'command', command: <node …gomad-statusline.js…>, padding: 0 } }`.
5. **Install case B — pre-existing settings.json with unrelated keys:**
   - Pre-create `settings.json` with `{ env: { FOO: 'bar' }, hooks: { SessionStart: [{ hooks: [{ type: 'command', command: 'echo hi' }] }] } }`.
   - Run setup. Assert `env.FOO === 'bar'` and `hooks.SessionStart[0].hooks[0].command === 'echo hi'` are still present, AND the new `statusLine` was added.
6. **Install case C — pre-existing third-party statusLine:**
   - Pre-create `settings.json` with `{ statusLine: { type: 'command', command: 'cat /tmp/foo' } }`.
   - Run setup. Assert `statusLine.command === 'cat /tmp/foo'` (untouched). Assert hook file IS still copied (we don't refuse the whole install — only the settings rewrite).
7. **Uninstall:**
   - From case A's resulting state, run `cleanup(tmpDir, {})`. Assert hook file is gone, `statusLine` key is gone from settings.json (or settings.json itself is `{}`).
   - From case B's resulting state, run `cleanup`. Assert `env.FOO === 'bar'` and the hooks block both survive.

Wire into `package.json` `scripts.test`: append ` && node test/test-statusline-install.js` to the `test` chain (same pattern as the other `test:*` entries get composed in `quality`/`test`).

**Verify:**
- `node test/test-statusline-install.js` exits 0.
- `npm test` includes the new test in its run.

**Done:** Test file created, all six assertions pass, `npm test` invokes it.
