# Quick Task 260512-vc3 — Rename gomad Claude Code hooks to .cjs

## Goal

In target projects whose `package.json` declares `"type": "module"`, the installed gomad hook crashes on every `UserPromptSubmit` with:

```
ReferenceError: require is not defined in ES module scope, you can use import instead
This file is being treated as an ES module because it has a '.js' file extension
and '/.../<target-project>/package.json' contains "type": "module".
To treat it as a CommonJS script, rename it to use the '.cjs' file extension.
                                                  → file:///.../.claude/hooks/gomad-agent-tracker.js:18
```

Root cause: `gomad-agent-tracker.js` (and `gomad-statusline.js`) use CommonJS (`require`, `module.exports`), but live with a `.js` extension under the host project's `.claude/hooks/` — so Node's module-type resolution is governed by the **host** project's `package.json`. Any host with `"type": "module"` will treat our hook as ESM and crash on first `require(...)`.

The user-visible symptom (`UserPromptSubmit hook error / Failed with non-blocking status code`) is Claude Code reporting the hook's non-zero exit; the hook is non-blocking, so the session continues, but the error renders on every prompt submission until fixed.

## Fix Strategy

Rename the two hook asset files from `.js` to `.cjs`. The `.cjs` extension forces Node to treat the file as CommonJS regardless of the host project's `package.json` `"type"` field. This is the migration Node itself recommends in the error message and is portable across Node versions / hosts.

Renaming the source assets requires synchronized updates to:

1. **The two source assets** under `tools/installer/assets/hooks/`
2. **`platform-codes.yaml`** — `source:` and `dest_name:` for both `statusline` and `agent_tracker`
3. **`_config-driven.js`** — installer logic identifies its own entries by checking `command.includes(dest_name)`. With `dest_name` flipped to `.cjs`, existing installs (which still have `.js` in their settings.json) would be misclassified as third-party. **Add legacy-`.js` cleanup logic** to `cleanupStatusline` and `cleanupAgentTracker` so that re-installing over an old `.js` install transparently migrates the user.
4. **`test/test-statusline-install.js`** — module-load paths and asserted file names. Add a Case H regression test that runs the tracker hook end-to-end under a synthetic `type: module` parent project to lock the fix.
5. **Allowlist fixtures** — `tools/fixtures/tarball-grep-allowlist.json` and `test/fixtures/orphan-refs/allowlist.json` reference the old paths.

## Migration Behavior (existing installs)

Existing users who already ran `gomad install` against the old version have:

- `<project>/.claude/hooks/gomad-agent-tracker.js` (broken under `type: module`)
- `<project>/.claude/hooks/gomad-statusline.js` (broken under `type: module`)
- Entries in `.claude/settings.json` that reference the `.js` filenames

The install path (`installToTarget` → `cleanup` → `installStatusline` / `installAgentTracker`) always runs `cleanup()` first. We extend `cleanupStatusline` and `cleanupAgentTracker` to also:

- Remove the legacy `.js` hook file if present at `<hooks_target_dir>/<basename-without-ext>.js`
- Strip any `statusLine.command` / `hooks.<Event>[].hooks[].command` referencing the `.js` filename, exactly as we already do for the current `dest_name`

This makes the migration zero-touch: `npx @xgent-ai/gomad install` self-heals existing broken installs in one command.
