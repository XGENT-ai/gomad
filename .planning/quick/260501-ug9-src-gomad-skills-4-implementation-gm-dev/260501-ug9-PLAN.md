## Current Problem

Both workflows currently write to sprint-status.yaml during story status transitions, but they lack explicit constraints preventing narrative/comment injection:

1. **gm-dev-story/workflow.md** — Steps 4 (mark in-progress) and 9 (mark review) update sprint-status.yaml. The step 9 instructions say "Save file, preserving ALL comments and structure including STATUS DEFINITIONS" which implies comments are expected/allowed. There is no explicit rule saying "only update status and last_updated, never add comments."

2. **gm-code-review/steps/step-04-present.md** — Section 6 "Sync sprint-status.yaml" updates the status after review. Again, no explicit constraint preventing comment addition.

The result: LLM agents running these workflows frequently append story completion narratives, review summaries, and issue notes directly into sprint-status.yaml, causing:
- Context window bloat (sprint-status.yaml grows unbounded)
- YAML format corruption (narrative text breaks parsing)
- Loss of sprint-status.yaml's purpose (it should be a compact status tracker, not a log)

- `{implementation_artifacts}/deferred-work.md` — already referenced by code-review step-04-present.md frontmatter and used for deferred findings
- `{implementation_artifacts}/known-issues.md` — not yet referenced but is the standard location for tracking known issues
