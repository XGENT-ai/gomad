       - Manifest CSV at `<installRoot>/_config/files-manifest.csv` records every emitted file with `install_root`, `path`, and metadata.
       - Manifest-driven cleanup: subsequent `gomad install` runs read the prior manifest, snapshot replaced files under `<installRoot>/_backups/<timestamp>/`, then write fresh outputs (cross-link in prose: `[upgrade recovery guide](../upgrade-recovery.md)`).
       - User overrides via `.customize.yaml` are preserved across reinstalls because the custom-file detector treats matching `.md` files as user-owned.
       - Show the v1.3 install layout in a code block:
         ```text
         <installRoot>/
         ├── _config/
         │   ├── agents/<shortName>.md
         │   ├── kb/<slug>/
         │   └── files-manifest.csv
         └── _backups/<timestamp>/
         ```

    5. `## The launcher-form slash command` — 4-5 paragraphs:
       - Three artifacts per persona:
         - Persona body: `<installRoot>/_config/agents/<shortName>.md` (the actual instructions).
         - Launcher stub: `.claude/commands/gm/agent-<shortName>.md` (the slash command target).
         - Slash command surface: `/gm:agent-<shortName>` (what the user types).
       - Runtime sequence: user types `/gm:agent-pm` → IDE loads launcher stub → stub instructs IDE to load persona body fresh → persona greets and lists capabilities.
       - Why launcher-form (not bundled): persona body changes on `gomad install` re-run; launcher stub stays stable. The user-facing slash command is decoupled from the persona content.
       - Cross-link in prose: `[agents reference](../reference/agents.md)` for the full persona catalog with slash commands.
       - Note that task skills (catalog at `[skills reference](../reference/skills.md)`) are loaded by the persona, not directly invokable as slash commands (D-09).

    6. `## How they fit together` — 2-3 paragraphs synthesis:
       - Walk through one concrete flow: developer runs `gomad install` (manifest-v2 lands files at v1.3 paths) → developer types `/gm:agent-pm` (launcher loads persona body) → persona drives the Planning phase (workflow) → output is a PRD draft consumed by the next phase.
       - Closing point: "the three pillars are independently testable but designed together — install lays files, launcher binds slash commands to those files, workflow gives them sequence."
