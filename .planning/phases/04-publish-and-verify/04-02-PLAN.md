<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| test runner → spawned npm subprocess | The e2e test invokes `npm pack`, `npm install <tarball>`, and the gomad CLI as subprocesses. Each subprocess reads from disk and writes to a tmp fixture under `os.tmpdir()`. |
| tmp fixture → host filesystem | The fixture lives in `os.tmpdir()` with a randomized name. A symlink attack or path-traversal in the install logic could let writes escape the fixture. |
| tarball lifecycle scripts → host | `npm install <tarball>` would normally run any `preinstall`/`install`/`postinstall` scripts in the tarball's package.json. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-07 | Tampering | tmp fixture path | mitigate | Fixture name uses `crypto.randomBytes(8).toString('hex')` (16 hex chars = 64 bits of entropy), making collision-or-guess attacks infeasible. `mkdirSync(fixtureDir, { recursive: true })` is the only filesystem write before the npm install step. |
| T-04-08 | Elevation of Privilege | npm install lifecycle scripts | mitigate | The e2e test passes `--ignore-scripts` to `npm install <tarball>` per Task 1. gomad has no install hooks today, but defense-in-depth blocks any future drift where a malicious dep-of-a-dep could run code. |
| T-04-09 | Information Disclosure | tmp dir leak after test failure | mitigate | `after()` hook is guaranteed to run by `node:test`. Cleanup uses `rmSync(dir, { recursive: true, force: true })` so partial-setup failures don't double-fault. |
| T-04-10 | Denial of Service | runaway test (e.g. infinite install loop) | mitigate | Each `execSync` call has an explicit `timeout` (60s for `npm pack`, 90s for `npm install`, 60s for the CLI run). Runner-level `--test-timeout=120000` is the outer ceiling. |
| T-04-11 | Spoofing | tarball substitution | accept | The test uses the tarball produced by `npm pack` in the same process, so there is no window for substitution. |
| T-04-12 | Repudiation | none | accept | Test runs locally and in CI; node:test reports failures with stack traces. |
</threat_model>
